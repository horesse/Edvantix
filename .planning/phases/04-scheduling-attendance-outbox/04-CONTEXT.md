# Phase 4: Scheduling — Attendance and Outbox - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Attendance is recorded for each student on each lesson slot. The `AttendanceRecord` aggregate enforces a unique constraint on `(LessonSlotId, StudentId)` — marking the same student twice updates the existing record rather than creating a duplicate. After each upsert an `AttendanceRecordedIntegrationEvent` is published to the `edvantix.scheduling.attendance-recorded` Kafka topic via the MassTransit transactional outbox already wired in Phase 3.

Payment logic, balance debit, and idempotent event consumption are Phase 5.

</domain>

<decisions>
## Implementation Decisions

### AttendanceRecord aggregate
- **D-01:** `AttendanceRecord` is a separate aggregate (not a child entity of `LessonSlot`) with its own repository — same isolation pattern as `LessonSlot`
- **D-02:** Unique constraint on `(LessonSlotId, StudentId)` — DB-level unique index; command handler does an upsert (insert or update existing record)
- **D-03:** Four statuses as a C# enum: `Present`, `Absent`, `Excused`, `Late`
- **D-04:** `CorrelationId` is a `Guid` (v7) generated by the server on initial `AttendanceRecord` creation; it is immutable — subsequent status updates reuse the same `CorrelationId`
- **D-05:** Same `SchedulingDbContext` — no separate DbContext needed

### API endpoints
- **D-06:** `PUT /slots/{slotId}/attendance/{studentId}` — per-student upsert, body contains `{ status }`. Returns 200 regardless of create vs. update
- **D-07:** `GET /slots/{slotId}/attendance` — returns list of `{ studentId, status, correlationId, markedAt }` for the slot; protected by `scheduling.view-schedule`

### Permissions
- **D-08:** New permission `scheduling.mark-attendance` added to `SchedulingPermissions` and seeded to Organizations. Both manager and teacher roles can hold this permission
- **D-09:** `GET /slots/{slotId}/attendance` reuses existing `scheduling.view-schedule` — no new permission needed

### Integration event and outbox
- **D-10:** Event published on **every upsert** (create + update) — not only on first mark. Payments must handle status changes (e.g. Present → Absent means debit may need reversal in Phase 5)
- **D-11:** Kafka topic name is auto-derived by Chassis `KebabCaseEndpointNameFormatter.SanitizeName` from the class name → `attendance-recorded-integration-event`. The original name `edvantix.scheduling.attendance-recorded` cannot be achieved without a Chassis override (accepted deviation, confirmed 2026-03-21). Phase 5 consumer MUST subscribe to `attendance-recorded-integration-event`.
- **D-12:** `AttendanceRecordedIntegrationEvent` lives in `Edvantix.Contracts` (same as `UserPermissionsInvalidatedIntegrationEvent`). Payload: `StudentId`, `LessonSlotId`, `SchoolId`, `Status` (string), `Timestamp` (DateTimeOffset), `CorrelationId` (Guid)
- **D-13:** Domain event `AttendanceRecordedEvent` raised inside the aggregate → `EventMapper` in Scheduling Infrastructure → `AttendanceRecordedIntegrationEvent` dispatched via outbox. Follows Organizations pattern exactly
- **D-14:** Outbox infrastructure already wired in Phase 3 (`AddEntityFrameworkOutbox<SchedulingDbContext>` + `UseEntityFrameworkOutbox`) — Phase 4 only adds the event type and mapper

### Claude's Discretion
- EF Core index strategy for the unique constraint (composite unique index vs. filtered)
- Whether `markedAt` is stored as a separate column or derived from EF Core audit fields
- Mediator domain event handler wiring (follows Phase 02-02 pattern: handlers in `Domain/EventHandlers/`)

</decisions>

<specifics>
## Specific Ideas

- CorrelationId is stable across status changes — Payments uses it as idempotency key. Regenerating on update would break Phase 5 deduplication
- `Status` in the event payload sent as string (not int) so Payments can deserialize without sharing the enum type
- Teacher can mark attendance because they are present in the classroom — restricting to manager-only would be impractical

</specifics>
