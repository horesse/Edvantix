## Summary

Phase 4 adds `AttendanceRecord` as a second aggregate in the Scheduling service and connects it to the MassTransit transactional outbox that was provisioned but unused in Phase 3. The implementation is almost entirely additive — no existing code needs modification beyond `SchedulingDbContext`, `SchedulingPermissions`, and `Extensions.cs`.

The domain event pipeline is fully established by Phase 2 (Organizations). Scheduling already has `IEventMapper`, `IEventDispatcher`, `EventDispatchInterceptor`, and `AddEntityFrameworkOutbox<SchedulingDbContext>` wired. Phase 4 supplies the concrete `EventMapper` in `Scheduling/Infrastructure/EventServices/EventMapper.cs`, registers it as `services.AddScoped<IEventMapper, EventMapper>()` and calls `services.AddEventDispatcher()` — exactly as Organizations does in its own `Extensions.cs`.

The primary implementation risk is the upsert pattern: EF Core change-tracking upsert (`FindAsync` → update or new entity) is preferred over `ExecuteUpdate` because it triggers the `SaveChanges` interceptor that dispatches domain events. Using `ExecuteUpdate` bypasses change tracking and therefore bypasses `EventDispatchInterceptor`, which means the domain event never fires and the outbox never receives the integration event.

**Primary recommendation:** Model the `MarkAttendanceCommandHandler` as a tracked-entity upsert: query for existing `AttendanceRecord` by `(LessonSlotId, StudentId)` using `FirstOrDefaultAsync`, call `MarkAttendance(status)` on the existing instance (or construct a new one), call `repository.Add` only for inserts, then `unitOfWork.SaveEntitiesAsync`.

### Pattern 1: AttendanceRecord Aggregate
**What:** `Entity`, `IAggregateRoot`, `ITenanted` — same interfaces as `LessonSlot`. Raises `AttendanceRecordedEvent` via `RegisterDomainEvent` in both the constructor (new) and `UpdateStatus` (upsert).
**When to use:** Any time attendance state changes (initial mark and subsequent updates).

```csharp
// Source: LessonSlot.cs constructor pattern + HasDomainEvents.RegisterDomainEvent
public sealed class AttendanceRecord : Entity, IAggregateRoot, ITenanted
{
    public Guid SchoolId { get; private set; }
    public Guid LessonSlotId { get; private set; }
    public Guid StudentId { get; private set; }
    public AttendanceStatus Status { get; private set; }
    public Guid CorrelationId { get; private set; }
    public DateTimeOffset MarkedAt { get; private set; }

    private AttendanceRecord() { } // EF Core

    public AttendanceRecord(
        Guid schoolId, Guid lessonSlotId, Guid studentId, AttendanceStatus status)
    {
        Guard.Against.Default(schoolId, nameof(schoolId));
        Guard.Against.Default(lessonSlotId, nameof(lessonSlotId));
        Guard.Against.Default(studentId, nameof(studentId));

        SchoolId = schoolId;
        LessonSlotId = lessonSlotId;
        StudentId = studentId;
        Status = status;
        CorrelationId = Guid.CreateVersion7(); // stable across updates (D-04)
        MarkedAt = DateTimeOffset.UtcNow;

        RegisterDomainEvent(new AttendanceRecordedEvent(this));
    }

    public void UpdateStatus(AttendanceStatus newStatus)
    {
        Status = newStatus;
        MarkedAt = DateTimeOffset.UtcNow;
        RegisterDomainEvent(new AttendanceRecordedEvent(this));
    }
}
```

### Pattern 2: Upsert via EF Core Change Tracking
**What:** Read existing record with `FirstOrDefaultAsync` (respects tenant query filter). Call `UpdateStatus` if found, or construct new and add. Persist via `unitOfWork.SaveEntitiesAsync`.
**When to use:** ALWAYS for upsert operations that must produce domain events. Do not use `ExecuteUpdate`/`ExecuteInsert` — they bypass `EventDispatchInterceptor`.

```csharp
// Source: CreateLessonSlotCommandHandler.cs + EventDispatchInterceptor.cs
public async ValueTask<Unit> Handle(MarkAttendanceCommand command, CancellationToken ct)
{
    var existing = await repository.FindBySlotAndStudentAsync(
        command.SlotId, command.StudentId, ct);

    if (existing is null)
    {
        var record = new AttendanceRecord(
            tenantContext.SchoolId, command.SlotId, command.StudentId, command.Status);
        repository.Add(record);
    }
    else
    {
        existing.UpdateStatus(command.Status);
    }

    await unitOfWork.SaveEntitiesAsync(ct);
    return Unit.Value;
}
```

### Pattern 3: Domain Event → Integration Event Pipeline
**What:** The domain event is raised inside the aggregate, collected by `EventDispatchInterceptor` after `SaveChanges`, dispatched via `IEventDispatcher.DispatchAsync` → `EventMapper.MapToIntegrationEvent` → `IBus.Publish` → outbox.
**When to use:** Any time an integration event must be published transactionally with the DB write.

**Full wiring (verified from source):**
1. `EventDispatchInterceptor.SavedChangesAsync` collects `IHasDomainEvents` entries with non-empty `DomainEvents`
2. `IDomainEventDispatcher.DispatchAndClearEvents` publishes each via Mediator `INotificationHandler<T>`
3. The handler calls `IEventDispatcher.DispatchAsync(domainEvent)`
4. `EventDispatcher` calls `IEventMapper.MapToIntegrationEvent` then `IBus.Publish((object)integrationEvent)`
5. Because `UseEntityFrameworkOutbox` is registered, `bus.Publish` writes to `OutboxMessage` in the same DB transaction
6. The outbox relay picks up the message and publishes to Kafka

**Two registrations required in `Extensions.cs`:**
```csharp
// Source: Organizations Extensions.cs lines 117-118
services.AddScoped<IEventMapper, EventMapper>();
services.AddEventDispatcher();
```

**What:** `HasIndex` with `IsUnique()` on `(LessonSlotId, StudentId)`.
**Discretion guidance:** A plain composite unique index (not filtered) is appropriate — `AttendanceRecord` has no soft-delete and the combination should be globally unique within the tenant.

### Anti-Patterns to Avoid
- **`ExecuteUpdate` for upsert:** Bypasses EF change tracking, so `EventDispatchInterceptor` never sees the entity, domain events are never raised, and the outbox never receives the integration event. Always use tracked-entity approach.
- **Generating new `CorrelationId` on update:** Breaks idempotency in Phase 5 Payments. The `CorrelationId` must be read from the existing record and preserved. It is a property on the aggregate that `UpdateStatus` does not touch.
- **Placing `AttendanceRecordedEvent` in `Domain/AggregatesModel/`:** The arch test `DomainEventTests.GivenDomainEvents_WhenCheckingDependencies_ThenShouldNotDependOnInfrastructure` bans infrastructure imports from domain events. Events in Organizations live in `Infrastructure/EventServices/Events/` (confirmed from source). Follow the same folder.
- **Sending `Status` as int in the integration event:** D-12 specifies `Status` as `string` so Payments can deserialize without sharing the enum type. Use `status.ToString()` in `EventMapper`.
- **Omitting `services.AddScoped<IEventMapper, EventMapper>()`:** The `EventDispatcher` will throw `NullReferenceException` at runtime because no mapper is registered. This was the only wiring gap in Scheduling before Phase 4.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Transactional event publication | Custom outbox table | `AddEntityFrameworkOutbox` (already wired) | Handles relay, deduplication, retry automatically |
| Event dispatch after save | Manual `Publish` in handler | `EventDispatchInterceptor` + `RegisterDomainEvent` | Interceptor fires after `SaveChanges` — correct transactional order |
| Idempotency key generation | Random Guid on every write | `Guid.CreateVersion7()` once at creation | Monotonic ordering + stable across updates (D-04) |
| Kafka topic creation | Manual broker config | `e.CreateIfMissing()` in `TopicEndpointConfigurator` | Already in Chassis `Extensions.cs` for consumers; producers auto-register topic name from message type name |

**Key insight:** The Kafka topic name is derived automatically from the message class name via `_formatter.SanitizeName(messageType.Name)` in `RegisterKafkaProducers`. The topic for `AttendanceRecordedIntegrationEvent` will be `attendance-recorded-integration-event` (KebabCase). To get the exact topic name `edvantix.scheduling.attendance-recorded` as specified in D-11, a custom `KafkaPublishFilter` or explicit producer registration with the correct topic name string is needed. **This is a critical finding — see Open Questions #1.**

## Common Pitfalls

### Pitfall 1: Kafka Topic Name Mismatch
**What goes wrong:** The auto-derived topic name from `KebabCaseEndpointNameFormatter.SanitizeName("AttendanceRecordedIntegrationEvent")` produces `attendance-recorded-integration-event`, not `edvantix.scheduling.attendance-recorded`.
**Why it happens:** `RegisterKafkaProducers` in Chassis `Extensions.cs` uses the class name, not a configurable topic name constant.
**How to avoid:** Check how `KafkaPublishFilter` routes messages, or verify if the producer registration accepts an override topic. Phase 4 plan must explicitly address this by either: (a) checking the `KafkaPublishFilter` for a topic-name override, or (b) accepting the auto-derived name and updating the D-11 topic string.
**Warning signs:** Build succeeds but Kafka producer sends to wrong topic; Phase 5 consumer never receives events.

### Pitfall 2: Arch Test Failure — ITenanted Without Query Filter
**What goes wrong:** `SchedulingTenantIsolationTests` fails the build if `AttendanceRecord` implements `ITenanted` but `SchedulingDbContext.OnModelCreating` does not register a `HasQueryFilter`.
**Why it happens:** The test scans the EF model for all `ITenanted` entity types and asserts each has a declared query filter.
**How to avoid:** Add `modelBuilder.Entity<AttendanceRecord>().HasQueryFilter(...)` immediately after the `LessonSlot` filter in `OnModelCreating`.
**Warning signs:** `just test` fails with `Entity AttendanceRecord implements ITenanted but has no HasQueryFilter`.

### Pitfall 3: Domain Event Not Dispatched
**What goes wrong:** Integration event never appears in outbox despite attendance being saved.
**Why it happens:** Either (a) `RegisterDomainEvent` not called in aggregate method, (b) `services.AddScoped<IEventMapper, EventMapper>()` not called in `Extensions.cs`, or (c) `services.AddEventDispatcher()` not called.
**How to avoid:** Verify all three registrations exist; unit-test `EventMapper.MapToIntegrationEvent` independently.
**Warning signs:** Attendance saved to DB but no row in `OutboxMessage` table.

### Pitfall 4: Upsert Race Condition
**What goes wrong:** Two concurrent requests for the same `(SlotId, StudentId)` both pass the `FindBySlotAndStudentAsync` null check and both try to insert, causing a unique constraint violation.
**Why it happens:** Standard "read then write" without database-level serialization.
**How to avoid:** The DB unique index on `(LessonSlotId, StudentId)` provides the safety net — the second insert will throw a `DbUpdateException` with a unique constraint violation. The command handler should catch this and retry as an update (or let the exception surface as a 409/422). Alternatively, use `ON CONFLICT DO UPDATE` via raw SQL, but this bypasses domain events.
**Warning signs:** `23505` unique constraint violation in Postgres logs under concurrent load.

### Pitfall 5: Arch Test — Domain Event in Wrong Folder
**What goes wrong:** `DomainEventTests.GivenDomainEvents_WhenCheckingDependencies_ThenShouldNotDependOnInfrastructure` fails if `AttendanceRecordedEvent` imports infrastructure types.
**Why it happens:** Domain events must not reference Infrastructure layer.
**How to avoid:** Keep `AttendanceRecordedEvent` free of infrastructure imports. It may only carry value-type fields (`Guid`, `string`, `DateTimeOffset`, enum). Place in `Infrastructure/EventServices/Events/` (same as Organizations — the namespace must NOT be the Infrastructure namespace; the arch test checks imports, not folder).

## Open Questions

1. **Kafka topic name — does auto-derived name match D-11?**
   - What we know: `RegisterKafkaProducers` in Chassis derives the topic name as `_formatter.SanitizeName(messageType.Name)` = `attendance-recorded-integration-event`.
   - What's unclear: Whether `KafkaPublishFilter` or any other mechanism allows overriding the topic string to `edvantix.scheduling.attendance-recorded`.
   - Recommendation: The planner should check `KafkaPublishFilter.cs` and if no override exists, decide whether to accept the auto-derived name or rename the integration event class to `AttendanceRecordedEvent` (which would produce `attendance-recorded`). Alternatively, the D-11 topic name may be aspirational and the auto-derived name is acceptable for v1.

2. **`markedAt` storage — separate column or derived?**
   - What we know: `DateTimeOffset` is the standard for all timestamps (SCH-10). `UpdateStatus` must update `MarkedAt` to reflect the time of the latest mark.
   - Recommendation: Store as a separate `timestamp with time zone` column (same as `StartTime`/`EndTime` in `LessonSlotConfiguration`). This is the simplest and most queryable option.
