# Edvantix

## What This Is

Edvantix — SaaS-платформа управления онлайн-школами. Одна инсталляция обслуживает множество школ (мультитенантность). Каждая школа управляет своим расписанием, студентами, преподавателями и платежами через единый интерфейс.

## Core Value

Менеджер школы может создать расписание группы, студент видит свои уроки и баланс, учитель видит свои занятия — всё в рамках одной школы с изолированными данными.

### Out of Scope

- Интеграция с платёжными системами (Stripe, ЮKassa и др.) — defer, не нужна в v1
- Программа/учебный план учителя — опциональная фича, не входит в этот майлстон
- Мобильное приложение — веб-первично
- Real-time уведомления — не в этом майлстоне

## Context

**Архитектура:** 3 отдельных микросервиса — Organizations, Payments, Scheduling. Каждый со своей БД и API.

**AuthN vs AuthZ:** Keycloak отвечает только за аутентификацию (выдаёт JWT). Авторизация — кастомная RBAC-система внутри Organizations service. Проверка происходит в каждом сервисе через обращение к Organizations (или кешированные данные).

**RBAC-модель:** Владелец школы → создаёт роли → вешает permissions → назначает роли пользователям. Permission концептуально = идентификатор CQRS-команды (например, `scheduling.create-slot`).

**Платёжная модель:** Пакетная — студент покупает N уроков. Уроки списываются по мере посещения. В v1 нет интеграции с процессингом, всё ручное.

**Frontend:** Next.js 16, pnpm монорепо (Turbo), Keycloakify для кастомизации Keycloak UI.

## Constraints

- **Tech stack**: .NET 10, Next.js 16, Keycloak, .NET Aspire — не менять
- **Multitenancy**: Данные разных школ строго изолированы на уровне каждого сервиса
- **Payments v1**: Только отображение и ручное управление, без процессинга

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Keycloak только для AuthN | Кастомная RBAC даёт гибкость (владелец сам создаёт роли), Keycloak AuthZ слишком жёсткий | — Pending |
| Permission = CQRS-команда | Прямое соответствие между правами и действиями в системе, легко проверять | — Pending |
| 3 отдельных микросервиса | Независимое масштабирование и деплой, чёткие bounded contexts | — Pending |
